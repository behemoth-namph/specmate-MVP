Sign-up — SRS

For SC-002 · Sign-up · AI drafted 18 m ago · skeleton: Starter · 5 sections

Overview

The Sign-up screen lets a new Acme team-member create an account and a Workspace in one step. It is the entry point for self-service onboarding; users arrive from a marketing CTA or a teammate's invite link.

Primary actions

  • Submit credentials — primary button posts to POST /api/v1/auth/sign-up; validates work email server-side; password must be 12+ chars with one number and one symbol.
  • Sign in instead — secondary link at the bottom for returning users; routes to /sign-in.
  • Marketing opt-in — checkbox defaults to opt-out (must be flipped to opt-in per GDPR — see annotation #2).

Data shown

The Acme brand mark, a welcome line, and a single Sign-up form. The form takes work email, password, and workspace name. Below: marketing footer with Privacy, Terms, copyright year.

Edge cases & empty states

  • Existing email → inline error "An account already exists for this email. Sign in instead?" with a link to /sign-in.
  • Weak password → inline meter showing weak / okay / strong; submit disabled until okay or higher.
  • Workspace slug collision → inline suggestion with a numeric suffix.
  • Network failure → banner "Couldn't reach the server. Try again."

Related Screens

  • Comes from: marketing landing (out-of-app), Sign-in's "Create an Acme account" CTA.
  • Goes to: SC-008 Onboarding · welcome on success, SC-001 Sign-in via secondary link.
Annotations #1 (password strength meter) and #3 (work-email validator) are open product questions tied to this screen — see Comments tab.

Tables written

The Sign-up flow writes to users, workspaces, and workspace_members in a single transaction. The first user is always inserted as the workspace owner.

users

emailtextunique, lowered, indexed
password_hashtextbcrypt cost 12
marketing_opt_inbooldefault false (GDPR)
created_attimestamp

workspaces

columntypenotes

slugtextunique, lowered, alphanumeric+dash

owner_user_iduuidfk → users.id
created_attimestamp

workspace_members

columntypenotes
iduuidpk
workspace_iduuidfk → workspaces.id
user_iduuidfk → users.id
roleenumowner / dev / client (v1: deferred)
joined_attimestamp

Indexes

  • users.email — unique, btree
  • workspaces.slug — unique, btree
  • workspace_members.workspace_id, user_id — composite unique

Notes

  • All three inserts run in a single BEGIN…COMMIT — partial state is impossible.
  • If the email is already taken, the entire transaction rolls back and the form shows an inline error (no race window).
  • Workspace slug collision is handled by appending a numeric suffix server-side before re-attempting the insert (max 5 retries).
